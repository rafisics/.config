## Context & Scope

Task 707 requests three convention changes to the literature extension:

1. **Co-located sources** — PDF/DJVU source files must live beside their converted markdown
   (in the same directory), and must be gitignored via `specs/literature/**/*.pdf` and
   `specs/literature/**/*.djvu` patterns.

2. **Content-aware chunking** — Replace the fixed 10-page-per-chunk approach with logical
   splitting at a 4,000-line threshold: detect chapter/section boundaries in the extracted
   text and split there; fall back to line-count splitting if no structure is found.

3. **Richer index.json schema** — Extend entries with: `author`, `title`, `year`,
   `doc_type` (paper/book/chapter/section), `source_format` (pdf/djvu), `parent_doc`
   (reference to parent document for chunk entries), `page_range`, and any other
   retrieval-useful fields.

The task also requires updating all affected documentation files.

#### Current Convert Flow (Summary)

1. Detect tool availability (`pdftotext`, `pdfinfo`, `djvutxt`).
2. For each target PDF/DJVU:
   a. Get page count via `pdfinfo`/`djvused`.
   b. Compute fixed chunks: `pages_per_chunk=10`. If total pages ≤ 10 → single file.
   c. Present proposed chunk boundaries to user via `AskUserQuestion`.
   d. Extract text per chunk using `pdftotext -f start -l end`.
   e. Write markdown file at `specs/literature/{basename}_p{start}-{end}.md`.
   f. Auto-extract keywords (word frequency) and summary (abstract or first sentences).
   g. Confirm metadata with user.
   h. Write index.json entry.

Key limitation: chunking is purely mechanical (page count). No analysis of chapter/section
structure in the text is performed.

### Current State Analysis Per Change

#### Change 1: Gitignore PDF/DJVU

- **Current**: No `.gitignore` patterns for literature sources. PDFs/DJVUs would be
  committed.
- **Proposed**: Add `specs/literature/**/*.pdf` and `specs/literature/**/*.djvu` patterns.
- **Impact**: Sources remain co-located (the layout is already compatible), but git will
  not track them. The `--convert` flow already scans for PDFs/DJVUs in the same directory,
  so no code changes are needed for the convert logic itself.
- **SKILL.md**: The "Standards Reference" section mentions chunking but says nothing about
  source co-location. The convert mode's file-finding logic (`find "$lit_dir" -name "*.pdf"`)
  already handles co-located files recursively. The scan output messages reference source
  files — these are unaffected.
- **Files affected**: `.gitignore` only (for the gitignore change). Documentation updates
  required in: `EXTENSION.md`, `README.md`, `SKILL.md` (Standards Reference section),
  `literature-organization.md`, `literature-agent.md`.

#### Change 2: Content-Aware Chunking

- **Current**: `pages_per_chunk=10`. For a 100-page book, this produces 10 chunks of 10
  pages each regardless of chapter boundaries. No text analysis occurs.
- **Proposed**: After extracting text, measure line count. If ≤ 4000 lines → single file.
  If > 4000 lines → detect chapter/section headings (e.g., lines matching `^Chapter`,
  `^Section`, `^\d+\.`, or markdown `## ` patterns) and split at those boundaries. If no
  headings are found, fall back to a line-count split (4000 lines per chunk).
- **Implementation location**: Convert Step 3b in `SKILL.md` (currently "Determine
  Chunking"). The page-based approach must be replaced with text-analysis approach.
- **DJVU limitation**: `djvutxt` extracts all text at once (no page-range support), so
  chunking for DJVU is already character/line based. This change makes PDF chunking
  consistent with DJVU.
- **User interaction change**: The AskUserQuestion at step 3c currently shows page ranges
  as chunk descriptions. With content-aware splitting, it should show detected section names
  or line ranges instead.
- **Output file naming**: Currently `{basename}_p{start_page}-{end_page}.md`. With
  content-aware splitting, naming should reflect logical structure:
  `{basename}_ch01_introduction.md`, `{basename}_sec02_syntax.md`, or
  `{basename}_part01.md` (if no headings detected, use sequential part numbers).
- **Files affected**: `SKILL.md` (Convert Step 3b, 3c, 3d, display summary table).
  Documentation updates in: `literature-agent.md`, `EXTENSION.md`, `README.md`,
  `literature-organization.md`.

#### Change 3: Richer Index Schema

- **Current schema fields**: `id`, `path`, `token_count`, `keywords`, `summary`
- **Fields already documented** (in `literature-organization.md`, advisory): `bib_key`,
  `authors`, `year`, `section`, `page_range`, `title`
- **New required fields** (from task description):
  - `author` — first author last name (string), or `authors` array
  - `title` — document title (string)
  - `year` — publication year (integer, nullable)
  - `doc_type` — `"paper"` | `"book"` | `"chapter"` | `"section"` (enum string)
  - `source_format` — `"pdf"` | `"djvu"` | `"manual"` (enum string, "manual" for hand-written markdown)
  - `parent_doc` — entry `id` of parent document for chunk/section entries (string, nullable)
  - `page_range` — source page range in original document (string like `"15-47"`, nullable)
- **Field to reconcile**: `literature-organization.md` uses `authors` (array) while the
  task says `author` (singular). Recommend `authors` (array) as the canonical field, with
  single-author entries using a one-element array.
- **Field to reconcile**: `bib_key` is already in the advisory schema; it should be
  retained as an optional field.
- **Impact on SKILL.md**: The convert step 3g (Update index.json) builds entries with only
  5 fields. It must be updated to prompt for or auto-detect the new fields. The auto-detect
  logic for `title` (from filename), `year` (from filename pattern `Author_Year_Title`),
  `doc_type` (default "paper" for flat files, "section" for chunked entries), `source_format`
  (from file extension), and `parent_doc` (from parent document slug for chunks) can be
  largely automated from context available during conversion.
- **Impact on `--index` mode**: Similarly, the Index step 4 (prompt user for metadata) and
  step 5 (write to index.json) must be updated.
- **Impact on validate mode**: The validate mode should check that required new fields are
  present (non-null) on each entry.
- **Files affected**: `SKILL.md` (Convert 3f/3g, Index 4/5, Validate Step 2), `literature-agent.md`
  (Index Schema section), `literature-organization.md` (Entry fields table), `EXTENSION.md`, `README.md`.

#### Chunking Algorithm for Change 2

```
1. Extract full text from source (pdftotext all pages, or djvutxt).
2. Split text into lines.
3. If line_count <= 4000 → single file (no chunking).
4. Else:
   a. Scan for structural headings:
      - PDF: lines matching /^(Chapter|CHAPTER)\s+\d+/, /^\d+\s+[A-Z]/, /^Part\s+[IVX\d]+/
      - Markdown: lines matching /^#{1,3}\s+/, /^={3,}/, /^-{3,}/
   b. If headings found:
      - Group consecutive content between headings into logical sections.
      - Merge adjacent small sections (< 500 lines) to stay close to 4000-line target.
      - Present detected section names to user for confirmation.
   c. If no headings found:
      - Split mechanically at 4000-line boundaries.
      - Name chunks sequentially: _part01.md, _part02.md, etc.
5. Output file naming:
   - Single-file: {basename}.md
   - Chapter chunks: {basename}/{sectionNN}_{slug}.md
   - Fallback (no headings): {basename}/{basename}_part{NN}.md
6. Ask user to confirm boundaries (existing AskUserQuestion step).
```

## Decisions

- Use `authors` (array) not `author` (singular) for the index schema, consistent with the
  existing advisory schema in `literature-organization.md`.
- Add `source_format: "manual"` for markdown files added directly via `--index` (no source
  PDF/DJVU).
- `parent_doc` uses entry `id` (not `path`) for cross-referencing, enabling renaming
  without schema corruption.
- The 4000-line threshold for chunking is treated as a target; logical boundaries take
  precedence (a chapter that spans 4200 lines stays intact rather than being split mid-chapter).
- The root `index.json` `token_budget` field remains (used by `literature-retrieve.sh`);
  the per-entry `token_count` continues to use `chars/4+20` formula.
- EXTENSION.md is the merge source for the CLAUDE.md section. Changes to the literature
  extension description in CLAUDE.md must be made via EXTENSION.md.
